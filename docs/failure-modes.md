Failure Mode Documentation
This document records every significant failure mode identified across SDF versions and the architectural response to each. It functions as the framework's regression record.
Each entry follows the same format: what broke, why it broke, and what architectural fix was introduced. Failure modes are not bugs — they are the development process. Every version of SDF exists because the prior version broke somewhere identifiable.


FM-001: Hard Gate / Preference Conflation Identified in: v1.0 | Resolved in: v1.5
Description: High preference scores were partially compensating for borderline hard gate performance. The boundary between non-negotiable requirements and weighted preferences existed conceptually but was not enforced architecturally. An input that should have been eliminated at the classification stage was surviving into scoring because its preference performance was strong enough to obscure the hard gate failure.
Resolution: Hard gates made strictly binary with zero tolerance for score compensation. Any input failing a hard gate is eliminated before entering the scoring phase regardless of projected preference performance. Hard requirements and preferences now operate in different stages and never interact.

FM-002: Implicit Weight Assignment Identified in: v1.5 | Resolved in: v2.0
Description: Weight assignments were inconsistent across evaluations. The same preference category carried different implicit weight depending on how the input was framed to the model. The model was determining what mattered rather than executing against weights the human had defined. This produced non-reproducible scoring — the same input evaluated twice could produce different weighted outcomes based on framing variation alone.
Resolution: Weight assignment formalized as explicit human input declared before the framework runs. Weights held constant across all inputs in a given evaluation set. The model no longer determines what matters — the human does, in advance, and the framework holds those weights fixed.

FM-003: Aggregate Score Masking Identified in: v2.0 | Resolved in: v2.1
Description: High overall scores were obscuring significant weakness in specific scoring categories. Risk tiering was producing misleading confidence signals — an input with a strong aggregate score but a critical category-level weakness was being classified at a higher confidence tier than its actual risk profile warranted. The weakness was being averaged away rather than surfaced.
Resolution: Explicit risk flag surfacing introduced at the category level. Category-level weaknesses appear in the output regardless of aggregate score. High aggregate performance can no longer mask material gaps in specific categories.

FM-004: Output Documentation Inconsistency Identified in: v2.1 | Resolved in: v2.2
Description: Output documentation quality varied with model behavior rather than following a defined structure. Rationale was inconsistent across evaluations — some outputs included detailed category-level reasoning while others produced summary-level conclusions without traceable logic. The inconsistency made it difficult to compare outputs across evaluations or identify where the framework logic had been applied correctly.
Resolution: Standardized output format defined and enforced. All outputs include hard gate results, category-level scores, risk flags, risk tier assignment, and recommendation rationale in a consistent structure regardless of model or decision domain.

FM-005: Model-Dependent Consistency Identified in: v2.2 | Resolved in: v2.3
Description: Framework behavior varied across LLM configurations. Output consistency was partially dependent on model characteristics rather than framework constraints alone. The same input evaluated against the same constraint architecture produced meaningfully different outputs depending on which model was running underneath — which meant consistency was a property of the model, not the framework.
Resolution: Framework validated across Claude, GPT-4, and Gemini. Constraint architecture confirmed model-agnostic. Where model behavior varied, the constraint layer absorbed the variance and produced consistent outputs. Consistency is now a property of the framework architecture, not the model.
FM-006: Uniform Gap Penalty Identified in: v2.3 | Resolved in: v2.4
Description: Gap penalties were applied uniformly regardless of whether the gap was realistically closeable within the evaluation timeline. A gap closeable before first contact with minimal effort was penalized identically to A gap closeable before first contact with minimal effort was penalized identically to a gap representing a fundamental capability deficit requiring sustained development work to close. This produced inaccurate risk assessments — inputs with minor closeable gaps were being scored more conservatively than their actual risk profile warranted, while inputs with fundamental domain mismatches were not being differentiated clearly enough.
Resolution: Gap ramp time estimation introduced as a required component of scoring analysis. Four tiers — Days, Weeks, Months, Different Domain — calibrate score penalty to realistic closing timeline. Days and Weeks gaps carry reduced penalty reflecting their closeability within a standard evaluation cycle. Months gaps carry standard penalty with a mandatory roadmap note identifying specific closing activity and timeline. Different Domain gaps carry full hard gap penalty. Gap evaluation now produces both a penalty and an evaluation survivability call.
FM-007: Asymmetric Preferred Scoring Identified in: v2.4 | Resolved in: v2.5
Description: The preferred requirement scoring rule was asymmetric. It correctly avoided penalizing absences — missing a preferred requirement carried minimal weight — but it did not reward presence. Strong preferred matches were being undervalued and had no mechanism to add positive signal to the output. An input with exceptional preferred requirement coverage scored identically to one with no preferred coverage, as long as hard requirements were met equally.
Resolution: Preferred requirement scoring expanded to two-sided. Absence remains at 0.25x weight maximum — no penalty. Presence now scores as a bonus of 0.25–0.5 per strong match, up to 1.0 maximum, stacked on top of the hard requirement baseline. Hard guardrail added and enforced: preferred bonuses cannot compensate for hard requirement failures. The two-sided rule correctly values both sides of preferred requirement coverage.

FM-008: Undifferentiated Investment Identified in: v2.5 | Resolved in: v2.6
Description: All inputs were receiving the same depth of analytical investment regardless of consequence magnitude or rarity. High-stakes, rare inputs were being evaluated with the same effort as routine, low-consequence inputs. This produced two distinct failure modes simultaneously — over-investment in low-value inputs wasted resources, and under-investment in high-value inputs produced unreliable outputs on decisions that mattered most.
Resolution: Formal investment tier classification introduced before analysis begins. Three tiers — Standard, Full, and Elite — calibrate analytical depth to consequence magnitude and input rarity. Standard for common inputs. Full for high-value inputs. Elite for rare or high-consequence inputs where the output materially changes trajectory. Investment tier is documented before analysis begins, can be upgraded mid-analysis if the input proves more valuable than initially assessed, and cannot be downgraded to skip uncomfortable findings. Evaluator Probability Statement added — written from the evaluator's perspective to assess likelihood of a favorable result based on output match, competitive pool, visible differentiators, and visible gaps.

FM-009: Implied Credential Surfacing Identified in: v2.6 | Resolved in: v2.7
Description: On high-consequence inputs, the strongest relevant credentials were sometimes present in the output template but only implied rather than explicitly surfaced. The framework was producing analytically correct outputs but the most important credentials for a specific evaluation context were buried, reordered away from first visibility, or left for the evaluator to connect independently. Strong credentials that should have been immediately visible were not earning the recognition they warranted because they required inference rather than direct reading.
Resolution: Within-Category Emphasis Check introduced for Elite tier evaluations. Runs after investment classification and before scoring. Identifies the two to three signals the evaluator will look for first and verifies whether those signals are explicitly present or only implied in the output template. Flags what needs surfacing through targeted addition or reorder before submission. Evidence Library added as a canonical reference defining the ceiling of what can be claimed. Source documents define the actual scope of evidenced capability. Framing guidance must remain truthful to actual capability performed — the check surfaces and makes explicit what is already evidenced, never adds what isn't.
